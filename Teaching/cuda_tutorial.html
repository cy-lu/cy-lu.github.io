CUDA Tutorial for Scientific Computing

This page introduces GPU computing from a computational physics perspective. The emphasis is on understanding why GPUs can accelerate structured numerical workloads, how CUDA organizes parallel execution, and how these ideas connect naturally to stencil-based solvers, finite-difference methods, and PDE-oriented scientific programming.

1. Why can a GPU be much faster than a CPU?

The central reason is architectural specialization. A CPU is optimized for low-latency execution, complex control flow, branch-heavy logic, and strong single-thread performance. A GPU is optimized for throughput: it devotes much more of its hardware budget to arithmetic units and parallel execution resources, which makes it especially effective when the same operation must be applied to many data elements. This is exactly the situation encountered in array-based scientific computing, stencil updates, matrix operations, and many structured PDE solvers.

Interpretation. The GPU is not universally faster. It is most effective when the workload is highly parallel, arithmetic-rich, and regular enough that many threads can execute similar instructions on different data.

When GPU acceleration is a natural fit

  • Large vector and matrix operations
  • Finite-difference and finite-volume structured-grid kernels
  • Stencil-based updates and iterative relaxation methods
  • Dense linear algebra and repeated array transforms

2. CUDA execution model

CUDA exposes the GPU as a parallel device on which a kernel is executed by many threads. These threads are organized hierarchically into threads, blocks, and grids. This hierarchy is fundamental because it determines both indexing and cooperation.

  • Thread: one execution instance
  • Block: a cooperating group of threads
  • Grid: the full collection of blocks launched for one kernel

In the simplest one-dimensional case, each thread computes a global index according to 

int i = blockIdx.x * blockDim.x + threadIdx.x;

and then uses that index to update one element of an array or one spatial point of a discretized field. In two dimensions, the same idea is extended with x and y indices. 

int i = blockIdx.x * blockDim.x + threadIdx.x;
int j = blockIdx.y * blockDim.y + threadIdx.y;
Numerical viewpoint. This model is particularly natural for scientific computing because one thread can often be mapped to one degree of freedom, one mesh point, one grid cell, or one output datum.

3. GPU memory hierarchy

Raw arithmetic throughput alone does not determine performance. For many CUDA kernels, the true limiting factor is memory traffic. The GPU memory hierarchy therefore plays a central role in optimization. Fast storage levels are smaller and closer to computation, while larger storage levels have higher latency.

Useful beginner-level interpretation

  • Registers: private to each thread and the fastest storage available
  • Shared memory: visible to threads in the same block and very important for stencil tiling
  • L1 / L2 cache: hardware-managed caching levels
  • Global memory: large device memory, but relatively slow compared with registers and shared memory

For stencil computations, performance often depends on whether neighboring data are repeatedly reloaded from global memory or successfully reused from shared memory and cache. This is why memory-aware implementation is often as important as the numerical formula itself.

4. Laplace stencil and Jacobi iteration

A natural scientific computing example for CUDA is the discrete Laplace operator. In two dimensions, the standard five-point stencil uses the four nearest neighbors of each interior point. This same pattern is also the basis of Jacobi relaxation for solving the Laplace equation.

Discrete 2D Laplace operator:

∇²ui,j ≈ ui+1,j + ui-1,j + ui,j+1 + ui,j-1 − 4ui,j
Jacobi update for ∇²u = 0:

ui,jnew = 1/4 [ui+1,jold + ui-1,jold + ui,j+1old + ui,j-1old]

Why this example is important

  • It connects CUDA directly to computational physics and PDEs.
  • It shows how one thread can update one grid point.
  • It motivates shared-memory tiling and structured memory reuse.
  • It provides a clean bridge toward more advanced methods such as FDTD and Poisson solvers.

Representative CUDA kernel structure

__global__ void jacobi2d(float* u_new, const float* u_old, int nx, int ny)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    int j = blockIdx.y * blockDim.y + threadIdx.y;

    if (i <= 0 || i >= nx - 1 || j <= 0 || j >= ny - 1) return;

    int c = j * nx + i;

    u_new[c] = 0.25f * (
        u_old[c + 1] +
        u_old[c - 1] +
        u_old[c + nx] +
        u_old[c - nx]
    );
}

This kernel is simple, but it already contains the essential elements of GPU scientific computing: thread indexing, structured data layout, spatial locality, and repeated array updates over a mesh.

5. Practical notes for beginners

  • Start from correctness. Always compare the GPU result against a CPU reference implementation.
  • Begin with a simple kernel. Do not optimize too early.
  • Think about data layout. In structured-grid methods, memory access patterns matter enormously.
  • Profile before optimizing. Bandwidth bottlenecks are common in stencil codes.
  • Use GPUs for the right workloads. Large, regular, data-parallel problems are the best entry point.
A strong learning sequence is: vector update → 1D stencil → 2D stencil → Jacobi solver → memory optimization. This progression keeps both the numerical meaning and the CUDA execution model clear.
