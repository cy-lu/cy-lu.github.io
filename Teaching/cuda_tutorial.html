CUDA Tutorial for Scientific Computing

This tutorial introduces GPU computing from a computational physics perspective. The focus is strictly on application: why GPUs accelerate structured numerical workloads, how memory hierarchy dictates performance, and how to implement stencil-based algorithms like the Laplace operator effectively.

1. Why GPUs Accelerate Computation

The core difference lies in architectural specialization. CPUs are optimized for low-latency execution and complex control flow, making them ideal for sequential logic. GPUs are designed for massive throughput, dedicating the majority of their hardware to arithmetic logic units (ALUs). This architecture excels when identical operations are applied to large datasets simultaneously.

Think of the CPU as a highly specialized single worker handling complex, dynamic tasks. The GPU, conversely, is a massive highly-coordinated legion, executing thousands of simple, identical tasks in parallel.

In scientific computing, this throughput-oriented design is the perfect match for structured-grid PDE solvers or stencil algorithms. A practical example is a Finite-Difference Time-Domain (FDTD) electromagnetic simulation, where GPU acceleration routinely yields order-of-magnitude speedups over multithreaded CPUs.

Left: CPU FDTD execution. Right: CUDA-accelerated FDTD execution demonstrating massive throughput gains.

Key takeaway: GPU acceleration is not a universal solution. It delivers maximum impact for data-parallel workloads characterized by high arithmetic intensity and regular memory access patterns.

2. CUDA Execution Model

CUDA structures parallelism hierarchically. Work is launched as a kernel, executed by thousands of threads. These threads are grouped into blocks, and blocks form a grid. In numerical grid solvers, one thread typically computes the update for one spatial mesh node.

Global memory indexing in 1D is defined as: 

int i = blockIdx.x * blockDim.x + threadIdx.x;

Extended to 2D for surface or cross-section modeling: 

int i = blockIdx.x * blockDim.x + threadIdx.x;
int j = blockIdx.y * blockDim.y + threadIdx.y;

3. GPU Memory Hierarchy

In numerical computing, performance is often bottlenecked by data movement, not math. Optimizing a CUDA kernel requires understanding the memory hierarchy to minimize latency and maximize bandwidth.

  • Registers: Thread-private, zero-latency storage.
  • Shared Memory: Block-private, user-managed cache. Highly effective for stencil tiling.
  • L1 / L2 Cache: Hardware-managed cache hierarchy.
  • Global Memory: High capacity, high latency. Minimizing transactions here is critical.
Practical impact: In stencil operations, true acceleration is achieved by ensuring neighboring mesh nodes are reused efficiently from registers or cache, rather than repeatedly fetched from global memory.

4. 1D Laplace Operator Benchmark

The 1D discrete Laplace operator serves as the foundational test for memory access strategies:

(∇²u)i ≈ ui+1 − 2ui + ui−1

This isolates memory-access profiling from the complexity of full PDE solvers. We evaluate four distinct implementations:

  • Naive: Direct global memory reads/writes.
  • LDG: Read-only cache targeting via __ldg().
  • TexObject: Hardware texture cache fetches.
  • Shared: Explicit shared memory tiling with halo exchange.
__global__ void ker_laplace_naive(float* y, const float* x, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i >= n) return;

  int prev = (i == 0) ? n - 1 : i - 1;
  int next = (i == n - 1) ? 0 : i + 1;

  y[i] = x[next] - 2.0f * x[i] + x[prev];
}

5. 2D Laplace Operator Benchmark

Moving to practical modeling, the 2D five-point stencil is ubiquitous in physics (diffusion, electrostatics, FDTD updating schemes):

(∇²u)i,j ≈ ui+1,j + ui−1,j + ui,j+1 + ui,j−1 − 4ui,j 
__global__ void ker_laplace2d_naive(float* y, const float* x, int nx, int ny) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  int j = blockIdx.y * blockDim.y + threadIdx.y;
  if (i >= nx || j >= ny) return;

  int im = (i == 0)      ? nx - 1 : i - 1;
  int ip = (i == nx - 1) ? 0      : i + 1;
  int jm = (j == 0)      ? ny - 1 : j - 1;
  int jp = (j == ny - 1) ? 0      : j + 1;

  int c  = j * nx + i;
  int xm = j * nx + im;
  int xp = j * nx + ip;
  int ym = jm * nx + i;
  int yp = jp * nx + i;

  y[c] = x[xp] + x[xm] + x[yp] + x[ym] - 4.0f * x[c];
}

Result Breakdown

Runtime scales linearly with the number of grid points, confirming this is fundamentally a memory-bound workload. The performance ranking remains rigid across all grid sizes: 

Naive ≈ LDG  <  TexObject  <  Shared
  • Naive vs. LDG: Identical performance. On modern architectures like the T4, the unified L1/L2 cache perfectly handles regular, row-major memory accesses. The explicit __ldg() instruction offers zero additional benefit.
  • Texture Object: Consistently slower. Texture caching provides no advantage for simple, aligned Cartesian stencils over modern L1 cache.
  • Shared Memory: The slowest method. For extremely lightweight kernels like a 5-point update, the computational overhead of block synchronization, halo loading, and index bookkeeping heavily outweighs the reduction in global memory transactions.
Engineering Reality: Explicit shared memory tiling is not a silver bullet. Always profile before adding complexity. For basic stencils on modern GPUs, hardware cache often wins.

7. The Evolution of Texture Memory

Historically, older CUDA architectures heavily relied on texture memory to accelerate 2D spatial locality workloads. The texture cache was uniquely optimized for spatial coherence, outperforming the rudimentary L1 caches of the time.

However, modern GPU memory architectures have highly unified and sophisticated cache hierarchies. As demonstrated in the benchmark above, standard global memory accesses (Naive) now match or exceed the performance of texture fetches for regular grids. Today, texture memory should be reserved for specific use cases: irregular access patterns, hardware-level interpolation (e.g., in graphics), or specialized image processing tasks, rather than standard PDE solvers.

8. Beyond Naive CUDA: Advanced Optimizations

When hardware caches handle basic stencils efficiently, further speedups in production-grade PDE solvers (e.g., 3D FDTD or CFD) are achieved by addressing the specific bottlenecks of memory bandwidth and hardware occupancy.

I. Memory Coalescing

GPU global memory is accessed via 32-byte or 128-byte transactions. If all threads in a Warp (32 threads) access a contiguous and aligned memory block, the hardware fulfills the request in a single transaction.

Application: In multiphysics simulations, avoid Array of Structures (AoS) (e.g., struct{Ex,Ey,Hz} grid[N]) as it results in non-contiguous access for individual components. Use Structure of Arrays (SoA) to ensure 100% coalesced transactions for every field component load.

II. Kernel Fusion

Every kernel launch requires writing intermediate results back to global memory and re-reading them in the next step. Moving data off-chip is significantly slower than the computation itself.

Application: Combine sequential operations—such as gradient computation, boundary conditions, and time integration—into a single massive kernel. By keeping intermediate values in registers, you eliminate off-chip memory round-trips and maximize the compute-to-memory ratio.

III. Thread Coarsening

Mapping one thread to one grid point is logical but can be inefficient due to indexing overhead.

Application: Assign a single thread to compute a small cluster of nodes (e.g., 1x4 points). This reduces total kernel launch and indexing overhead. More importantly, it allows for register reuse: shared boundary data between the 4 points can be loaded once and kept in zero-latency registers, effectively bypassing even the L1 cache while increasing Instruction-Level Parallelism (ILP).
