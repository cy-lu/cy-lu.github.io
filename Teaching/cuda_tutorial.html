CUDA Tutorial for Scientific Computing

This tutorial introduces GPU computing from a computational physics perspective. The focus is strictly on application: why GPUs accelerate structured numerical workloads, how memory hierarchy dictates performance, and how to implement stencil-based algorithms like the Laplace operator effectively.

1. Why GPUs Accelerate Computation

Architectural specialization is the key. CPUs are optimized for low-latency execution and complex control flow, making them ideal for sequential logic. GPUs are designed for massive throughput, dedicating the majority of their hardware to arithmetic logic units (ALUs). This architecture excels when identical operations are applied to large datasets simultaneously.

In scientific computing, this throughput-oriented design is the perfect match for structured-grid PDE solvers. For example, in Finite-Difference Time-Domain (FDTD) electromagnetic simulations, GPU acceleration routinely yields speedups of 20x to 100x compared to optimized CPU implementations.

2. CUDA Execution Model

CUDA structures parallelism hierarchically. Work is launched as a kernel, executed by thousands of threads. Threads are grouped into blocks, and blocks form a grid. In numerical grid solvers, one thread typically computes the update for one spatial mesh node. 

// 1D Global Indexing
int i = blockIdx.x * blockDim.x + threadIdx.x;

// 2D Global Indexing
int i = blockIdx.x * blockDim.x + threadIdx.x;
int j = blockIdx.y * blockDim.y + threadIdx.y;

3. GPU Memory Hierarchy

Performance in numerical computing is often limited by data movement, not arithmetic logic. Optimizing a CUDA kernel requires managing the memory hierarchy to minimize latency and maximize bandwidth.

  • Registers: Thread-private, zero-latency storage for local variables.
  • Shared Memory: Block-private, user-managed cache. Crucial for stencil tiling.
  • Global Memory: High capacity but high latency. Minimizing transactions here is the primary optimization goal.

8. Beyond Naive CUDA: Advanced Optimizations

When hardware caches handle basic stencils efficiently, further speedups in production-grade PDE solvers (e.g., 3D FDTD or CFD) are achieved by addressing the specific bottlenecks of memory bandwidth and hardware occupancy.

I. Memory Coalescing

GPU global memory is accessed via 32-byte or 128-byte transactions. If all threads in a Warp (32 threads) access a contiguous and aligned memory block, the hardware fulfills the request in a single transaction.

Application: In multiphysics simulations, avoid Array of Structures (AoS) (e.g., struct{Ex,Ey,Hz} grid[N]) as it results in non-contiguous access for individual components. Use Structure of Arrays (SoA) to ensure 100% coalesced transactions for every field component load.

II. Kernel Fusion

Every kernel launch requires writing intermediate results back to global memory and re-reading them in the next step. Moving data off-chip is significantly slower than the computation itself.

Application: Combine sequential operations—such as gradient computation, boundary conditions, and time integration—into a single massive kernel. By keeping intermediate values in registers, you eliminate off-chip memory round-trips and maximize the compute-to-memory ratio.

III. Thread Coarsening

Mapping one thread to one grid point is logical but can be inefficient due to indexing overhead.

Application: Assign a single thread to compute a small cluster of nodes (e.g., 1x4 points). This reduces total kernel launch and indexing overhead. More importantly, it allows for register reuse: shared boundary data between the 4 points can be loaded once and kept in zero-latency registers, effectively bypassing even the L1 cache while increasing Instruction-Level Parallelism (ILP).
