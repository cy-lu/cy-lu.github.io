1. Why can a GPU be much faster than a CPU?

The short answer is that a GPU is designed to execute a very large number of similar arithmetic operations in parallel. A CPU is optimized for low-latency execution, complex control logic, branch prediction, large caches, and strong single-thread performance. A GPU instead sacrifices some of that flexibility in exchange for massive throughput.

CPU mindset

Few powerful cores, strong control logic, large cache hierarchy, excellent for sequential logic, irregular branching, and latency-sensitive tasks.

GPU mindset

Many lightweight execution lanes, high arithmetic throughput, optimized for data-parallel workloads where the same operation is applied to many elements.

A useful mental picture

Suppose we want to update ten million grid points using the same finite-difference formula. On a CPU, a small number of strong cores repeatedly perform the update. On a GPU, thousands of lightweight threads can cooperate so that many grid points are updated at the same time. If the operation is regular and memory access is reasonably organized, the GPU can achieve much higher throughput.

Important: a GPU is not automatically faster for every problem. Small problems, branch-heavy logic, irregular memory access, or workloads dominated by data transfer may show little gain or may even run slower.

2. CUDA execution model

In CUDA, you write a kernel, which is a function executed by many threads in parallel on the GPU. Threads are organized hierarchically:

Thread: one execution instance
Block: a cooperating group of threads
Grid: all blocks launched for one kernel call
Kernel launch: CPU asks GPU to execute the grid
Global index in 1D:
i = blockIdx.x × blockDim.x + threadIdx.x

That formula is central. It tells each thread which data element it is responsible for. In scientific computing, this often means one thread updates one array element, one grid point, or one cell in a structured mesh. 

__global__ void add_kernel(float* y, const float* x, int n)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) {
        y[i] = 2.0f * x[i];
    }
}

The CPU prepares the data, launches the kernel, and then the GPU executes many threads concurrently. This model is especially natural for vector operations, matrix-style updates, and stencil computations.

3. CUDA memory model

Performance is not determined only by arithmetic. Very often the real bottleneck is memory movement. A useful beginner-level picture is the following:

Global memory

Large but relatively slow. All threads can access it. Most arrays start here.

Shared memory

Small and much faster. Shared within one block. Very useful for stencil tiling.

Registers

Very fast private storage for each thread. Best for temporary local variables.

For numerical kernels such as Laplace stencils, the key question is often: how many times do I reload neighboring values from global memory? When many nearby threads need overlapping data, shared memory can reduce redundant loads and improve effective bandwidth.

4. First numerical example: the 1D discrete Laplace operator

A clean first example is the discrete second derivative in one dimension:

(∇²u)i ≈ ui+1 − 2ui + ui−1

This is not yet a full PDE solver. It is a stencil operator. It is ideal for teaching because the arithmetic is simple, the memory access pattern is clear, and the parallel structure maps naturally onto CUDA. 

__global__ void laplace1d(float* y, const float* x, int n)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i >= n) return;

    int prev = (i == 0) ? n - 1 : i - 1;
    int next = (i == n - 1) ? 0 : i + 1;

    y[i] = x[next] - 2.0f * x[i] + x[prev];
}

Why this example is pedagogically useful

  • It introduces thread indexing without too much code complexity.
  • It shows the role of neighboring data in stencil methods.
  • It provides a natural path to discuss cache behavior and shared memory.
  • It scales directly into 2D and 3D finite-difference formulations.

5. From 1D to 2D: Laplace operators and PDE solvers

In two dimensions, the standard five-point discrete Laplace operator becomes:

(∇²u)i,j ≈ ui+1,j + ui−1,j + ui,j+1 + ui,j−1 − 4ui,j

This is the foundation for many scientific and engineering solvers, including diffusion, electrostatics, steady-state temperature fields, and pressure Poisson equations in CFD contexts. 

__global__ void laplace2d(float* out, const float* in, int nx, int ny)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    int j = blockIdx.y * blockDim.y + threadIdx.y;

    if (i >= nx || j >= ny) return;

    int ip = (i == nx - 1) ? 0 : i + 1;
    int im = (i == 0) ? nx - 1 : i - 1;
    int jp = (j == ny - 1) ? 0 : j + 1;
    int jm = (j == 0) ? ny - 1 : j - 1;

    int c = j * nx + i;

    out[c] = in[j * nx + ip]
           + in[j * nx + im]
           + in[jp * nx + i]
           + in[jm * nx + i]
           - 4.0f * in[c];
}

Operator vs equation

There is an important distinction here. Applying the stencil once gives the discrete operator. Solving the Laplace equation means we iteratively update the field until it satisfies the governing equation subject to boundary conditions.

For ∇²u = 0, a Jacobi update in 2D is:

unewi,j = 1/4 [uoldi+1,j + uoldi−1,j + uoldi,j+1 + uoldi,j−1] 
__global__ void jacobi2d(float* u_new, const float* u_old, int nx, int ny)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    int j = blockIdx.y * blockDim.y + threadIdx.y;

    if (i <= 0 || i >= nx - 1 || j <= 0 || j >= ny - 1) return;

    int c = j * nx + i;
    u_new[c] = 0.25f * (u_old[c + 1] + u_old[c - 1] + u_old[c + nx] + u_old[c - nx]);
}

This is the natural bridge from CUDA basics into scientific GPU computing. It connects data-parallel programming to a real PDE solver structure.

6. Practical advice for beginners

  • Start from correctness. Always compare GPU results against a CPU reference.
  • Keep the first kernel simple. Do not begin with advanced optimizations.
  • Think in data layout. In stencil codes, memory access is often as important as arithmetic.
  • Profile before optimizing. Do not guess where the bottleneck is.
  • Use CUDA for the right class of problems. Large, structured, repetitive arithmetic is the best entry point.
A strong teaching sequence is: vector update → 1D stencil → 2D stencil → Jacobi solver → performance tuning. That path lets students understand both the numerical meaning and the GPU execution model at the same time.
